## 1. THE 100-POINT AUDIT MANDATE

You must systematically assess the codebase against the following 4 domains, totaling 100 explicit checking criteria. Evaluate and account for every single one of these vectors:

### DOMAIN 1: TRUST ARCHITECTURE & SECURE GATEKEEPING (25 Points)
1. Server-Side Session Enforcement: Are session validations handled explicitly on the server layer for every mutated path?
2. DevTools Bypass Countermeasures: Can a client intercept the UI state via developer consoles to manipulate backend flags?
3. Exposure of Client-Side Secrets: Do environment variables leak into the client bundle prefix (e.g., NEXT_PUBLIC_)?
4. Route-Level Role Authorization: Is multi-tenant RBAC (Role-Based Access Control) verified securely at the routing middleware layer?
5. Public Route Boundary Hardening: Are public-facing endpoints explicitly decoupled from private internal application pathways?
6. Strict Database Parameterization: Are raw string concatenations completely absent to prevent SQL injection?
7. Serialization & Pickling Vectors: Does the app utilize unsafe deserialization engines (like Python's 'pickle') that permit Arbitrary Code Execution?
8. LLM Prompt Injection Defenses: If system prompts exist, are user inputs structurally isolated from executive system instructions?
9. Input Sanitization & Type Enforcement: Are incoming payloads forced through structural type engines (e.g., Zod, Joi, Valibot)?
10. Cross-Site Scripting (XSS) Mitigation: Are dynamically rendered HTML segments safely escaped, sanitized, or fully blocked?
11. Cross-Site Request Forgery (CSRF) Hardening: Are mutating mutations (POST/PUT/PATCH/DELETE) protected by token verifications?
12. Security Headers Implementation: Are explicit Content Security Policies (CSP), HSTS, and X-Frame-Options configured?
13. CORS Policy Restriction: Is the Access-Control-Allow-Origin header locked down tightly to explicitly verified origins?
14. Raw Object Reference (IDOR) Hardening: Can guessing a sequential resource ID allow an unauthorized tenant to view sibling data?
15. Cryptographic Hashing Standards: Are sensitive user fields (passwords, PII) hashed with strong salts (e.g., Argon2, bcrypt)?
16. JWT/Token Expiry Architecture: Are authorization tokens structurally short-lived with isolated, secure refresh lifecycles?
17. Rate-Limiting & DDOS Shields: Do public endpoints feature token-bucket throttlers to prevent rapid, programmatic exhaustion?
18. File Upload Sanitization: Are file-upload streams scanned for MIME-type spoofing, directory traversal vectors, and extreme file sizes?
19. Broken Object Level Authorization: Does the app ensure that the user owns the specific object instance they are altering?
20. Sensitive Data in Logs: Does the application inadvertently stringify or print user credentials or auth tokens to console streams?
21. Mass Assignment Vulnerabilities: Can a client send extra JSON fields to implicitly overwrite administrative database columns?
22. Unvalidated Redirects/Forwards: Can an open URL parameter twist internal redirect flows toward external phishing targets?
23. Directory Traversal Checks: Are relative file paths safely stripped out of user input strings to prevent reading server configuration files?
24. Session Fixation Defenses: Does the auth core cycle and regenerate the primary session identifiers immediately upon a login event?
25. Secure Cookie Directives: Are all persistent cookies locked with HttpOnly, Secure, and SameSite=Strict/Lax parameters?
