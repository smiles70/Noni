# Role & Objective
You are an expert Principal Software Architect and Static Analysis Agent. Your task is to perform an aggressive, 100-point "AI Slop Audit" on this codebase. "AI Slop" in an app-building SaaS context refers to unoptimized, hallucinated, overly verbose, or structurally unsound code generated by LLMs that prioritizes "first-try execution" over long-term maintainability, security, and performance.

# Phase 1: Repository Discovery & Context Gathering
Before evaluating the checklist, run necessary discovery actions to understand the project structure. Execute or simulate the following steps:
1. Scan the root directory to map the tech stack, languages, and frameworks used.
2. If available, hook into tools like `Understand Anything` to view the business logic domain layers and dependency graphs rather than just a raw file tree.
3. Analyze package manifests (e.g., `package.json`, `requirements.txt`, `go.mod`) for dependency bloat.
4. Locate the core components, API routers, and state management files.

Once discovery is complete, evaluate the codebase against the following 100-Point Audit Checklist.

# Phase 2: The 100-Point AI Slop Checklist

## 1. Code Architecture & Redundancy (Points 1-20)
1. Are there monolithic components/files exceeding 500 lines that could easily be modularized?
2. Is the exact same helper logic duplicated across multiple files instead of utilizing a shared utility?
3. Are there vestigial, commented-out LLM alternative implementations left in the code?
4. Do functions contain passive "placeholder" code or `// TODO: implement later` comments left by an AI?
5. Are variable and function names overly verbose or generic (e.g., `processData`, `handleButtonClick2`)?
6. Are there unused imports or dead dependencies cluttering the top of core files?
7. Is code mixing multiple architectural patterns (e.g., mixing MVC with clean architecture arbitrarily)?
8. Are state updates causing unnecessary, cascading re-renders in UI components?
9. Is configuration hardcoded inside functions instead of using an `.env` or central config file?
10. Are there multiple conflicting fetch libraries or HTTP clients used across different files?
11. Is the file structure deeply nested or disorganized, indicating sequential generation without structural planning?
12. Do functions accept a massive, un-typed generic `any` or `dict` object instead of strict schemas/interfaces?
13. Are there inline styles or random CSS frameworks mixed together within the same UI view?
14. Is business logic directly injected into UI rendering code instead of decoupled custom hooks/controllers?
15. Are there orphaned routes or API endpoints that aren't hooked up to any working UI or logic?
16. Are database models missing proper relationships, forcing the application code to stitch data together manually?
17. Are there microservices or serverless functions that repeat the exact same middleware setups explicitly?
18. Does the code contain redundant conditional blocks (e.g., checking the same variable twice sequentially)?
19. Are error messages generic defaults (e.g., `"Something went wrong"`) instead of context-specific errors?
20. Is asynchronous code missing proper synchronization, running the risk of race conditions?

## 2. Robustness, Type Safety & Validation (Points 21-40)
21. Are type definitions relying heavily on `any` (TypeScript) or missing type hints entirely (Python)?
22. Is incoming API payload validation missing or bypassed using basic type-casting?
23. Do database queries lack sanitization or parameterized inputs?
24. Are network requests missing explicit timeout thresholds, risking hung processes?
25. Is there a complete absence of unit test suites (`.test.ts`, `test_*.py`) for newly generated features?
26. Are edge cases (e.g., `null`, `undefined`, empty string, negative numbers) explicitly handled in utility code?
27. Are API error blocks missing `try/catch` or equivalent recovery mechanisms?
28. Does the frontend application crash entirely when an upstream API returns a 500 error?
29. Are form inputs missing client-side or server-side validation constraints?
30. Is regex code copy-pasted without documentation, presenting catastrophic backtracking risks?
31. Are environment variables accessed directly throughout the codebase without a validation/parsing layer?
32. Do data migrations lack rollback paths or down-scripts?
33. Are component props untyped or missing default fallbacks?
34. Are floating-point calculations used for financial or precision-critical data?
35. Is local storage or session storage read/write logic missing serialization error handling?
36. Are race conditions handled when multiple rapid UI actions occur (e.g., double-clicking a submit button)?
37. Are native array operations used inefficiently over huge datasets where hash maps would be `O(1)`?
38. Do third-party API payloads lack runtime schema validation (e.g., Zod, Pydantic)?
39. Is the code missing proper logging levels (`info`, `warn`, `error`), relying instead on standard print/console statements?
40. Are WebSocket connections missing automated re-connection and heartbeat logic?

## 3. Performance & Infrastructure Optimization (Points 41-60)
41. Are database queries executing inside loops (the N+1 query problem)?
42. Are database tables missing critical indexes on foreign keys or high-frequency query columns?
43. Are massive datasets pulled down to the client-side to be filtered, instead of using server pagination?
44. Are static assets served directly from the app server rather than a optimized cloud bucket or CDN?
45. Is code splitting or lazy-loading absent on heavy routes or complex UI modules?
46. Are third-party scripts loaded synchronously, blocking the critical rendering path?
47. Do API responses lack caching headers (`Cache-Control`) for static or slowly changing data?
48. Is polling used continuously where WebSockets or SSE (Server-Sent Events) would be significantly more efficient?
49. Are image assets unoptimized, lacking modern formatting (e.g., WebP) or responsive sizing?
50. Are heavy calculations running on the main UI thread instead of being memoized or offloaded?
51. Does the backend block thread execution with synchronous I/O operations?
52. Are database connection pools unconfigured, risking connection exhaustion under load?
53. Is memory usage unbounded when processing large file uploads or streams?
54. Are infinite loop protections absent in recursive functions or event listeners?
55. Are expensive UI operations (e.g., window scrolling, search typing) missing debounce or throttle logic?
56. Are CSS animations forcing browser layout repaints instead of utilizing GPU-accelerated transforms?
57. Is the bundle size bloated due to importing entire libraries when only a single function is used?
58. Are service workers or caching strategies non-existent for offline-first resilience?
59. Does the app fail to implement graceful degradation when user bandwidth is restricted?
60. Are background worker queues absent for long-running processes, causing HTTP requests to timeout?

## 4. Security, Auth & Compliance (Points 61-80)
61. Are sensitive API secrets, keys, or passwords committed anywhere in the git history?
62. Is JWT authentication missing token expiration, structure validation, or proper signature verification?
63. Are CORS policies configured to wildcard (`*`) in production instead of specific whitelisted origins?
64. Is Role-Based Access Control (RBAC) missing server-side enforcement on protected endpoints?
65. Are user sessions lacking secure, HTTP-only, SameSite cookie configurations?
66. Is sensitive user data (PII) logged in cleartext inside application logs?
67. Are passwords hashed using weak or outdated algorithms (e.g., MD5, SHA1) instead of Argon2/Bcrypt?
68. Are API endpoints vulnerable to basic scraping due to missing rate-limiting middleware?
69. Is sensitive information exposed via verbose database error messages returned directly to the client?
70. Are file upload endpoints missing file type, extension, and magic number verification?
71. Is Cross-Site Scripting (XSS) possible via unescaped innerHTML injections?
72. Are ID tokens used interchangeably with Access tokens in the authorization flow?
73. Are security headers (e.g., CSP, X-Content-Type-Options, HSTS) missing from server configurations?
74. Is authorization checked only at the UI layer, leaving raw API endpoints completely public?
75. Is database access using an over-privileged root/admin user instead of a scoped application user?
76. Are password reset tokens predictable, long-lived, or lacking single-use validation?
77. Is encryption-at-rest missing for highly sensitive database fields?
78. Are multi-tenant SaaS data bounds enforced lazily via application logic instead of strict database-level scoping?
79. Are exposed API endpoints lacking explicit security audits or OpenAPI specification guardrails?
80. Are dependencies plagued by known CVEs because packages have never been updated or audited?

## 5. UI/UX Polish & Accessibility (Points 81-100)
81. Do interactive elements completely lack loading indicators or skeleton states during network fetching?
82. Are UI components unresponsive or visually broken on mobile viewports or varying screen sizes?
83. Is keyboard navigation broken (e.g., missing focus outlines, element tab-indexing out of order)?
84. Are image elements missing mandatory `alt` tags for screen readers?
85. Is text contrast ratio too low, violating basic WCAG accessibility guidelines?
86. Are toast notifications or error banners missing automatic dismissal or manual close capabilities?
87. Do forms clear all user input data completely when a single validation error occurs on submit?
88. Are layout shifts (CLS) prominent during data hydration or lazy image loading?
89. Are fonts unoptimized or jumping visually during page rendering (FOIT/FOUT)?
90. Are tooltips or dropdown menus clipping outside the viewable browser window edge?
91. Does the app lack empty states for lists, tables, or dashboards when no data is returned?
92. Are modal overlays missing an easy way to escape (e.g., clicking outside the modal or pressing `Esc`)?
93. Is the typography inconsistent, using varying weights, sizes, and tracking without a strict design token system?
94. Are action targets (buttons, links) too small or cramped for reliable mobile touch targets?
95. Are localized string formats hardcoded, making future multi-language internationalization difficult?
96. Do state transitions or route changes feel jarring due to an absolute lack of smooth UI transitions?
97. Are hover and active interactive states missing from buttons and links?
98. Does the application fail to persist minor UI state configurations (e.g., sidebar collapse) across page refreshes?
99. Are native browser elements (like date pickers or select dropdowns) styled so heavily that they break on specific browsers?
100. Is there no dark mode/light mode awareness, leading to unreadable text elements when systems toggle themes?

# Phase 4: Final Grading Formula
Conclude the entire audit by calculating a final grade for the repository. Start with 100 points and subtract points based on the severity of the findings:
* **High Severity:** Deduct 4 points
* **Medium Severity:** Deduct 2 points
* **Low Severity:** Deduct 1 point

**Final Grade Scales:**
* **90 - 100:** Grade A (Production Ready - Clean Engine)
* **80 - 89:** Grade B (Minor Slop - Needs Refactoring)
* **70 - 79:** Grade C (Significant Slop - High Maintenance Debt)
* **Under 70:** Grade F (Slop Bucket - Highly Unstable)

Provide a brief, concluding architectural summary regarding the maintainability of this repository.
