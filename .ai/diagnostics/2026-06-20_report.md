## 3. `pattern-drift` — `DebugAuth` uses runtime `IS_DEV` instead of compile-time guard

**Confidence:** confirmed

**Evidence:**
- `frontend/src/App.tsx:37-60` defines `DebugAuth` component.
- It is conditionally rendered at line 143: `{IS_DEV && <DebugAuth />}`.
- `IS_DEV` is imported from `frontend/src/lib/env.ts` (runtime check: `import.meta.env?.DEV !== undefined ? import.meta.env.DEV : true`).

**Impact:** Debug UI could surface in production under misconfiguration; color violates accessibility contract.
**File/Lines:** `frontend/src/App.tsx:37-60`, `frontend/src/lib/env.ts`.

## 4. `latent-bug-suspected` — `forceRefresh` counter oscillation risk

**Confidence:** possible

**Evidence:**
- `frontend/src/auth/AuthProvider.tsx` (lines not read in this pass, but 720 Assessment P8 documents): "`forceRefresh` counter could oscillate — no debounce."
- Auth state machine transitions: `TRANSIENT_ERROR` -> retry -> `AUTHENTICATING` -> `READY`/`REJECTED`.
- If a transient error resolves and immediately re-triggers, rapid state oscillation could cause re-render loops.

**Impact:** Motion/spatial instability per geragogy §7; CPU churn.
**File/Lines:** `frontend/src/auth/AuthProvider.tsx` (forceRefresh logic).

## 5. `latent-bug-suspected` — `TRANSIENT_ERROR` destroys spatial stability with full reload

**Confidence:** confirmed

**Evidence:**
- 720 Assessment P12: "`TRANSIENT_ERROR` shows 'retry = full reload' — loses app state."
- "Spatial stability destroyed. Geragogy §4: 'Element positions must remain stable across states'."
- `frontend/src/api/client.ts` (not fully read, but 720 assessment confirms): retry logic for GETs falls back to full page reload on persistent transient errors.

**Impact:** User loses all context on network blip; violates cognitive-safety-first principle.
**File/Lines:** `frontend/src/api/client.ts` (retry logic), `frontend/src/auth/AuthProvider.tsx` (error handling).

## 6. `latent-bug-suspected` — Post-signout Bearer remains valid until JWT expiry

**Confidence:** confirmed

**Evidence:**
- 720 Assessment P4: "After `signOut()`, old Bearer remains valid until JWT expiry."
- `backend/api/deps.py` verifies token per-request but has no revocation list.
- `backend/services/auth_provider.py` Clerk provider does not call Clerk revocation API on signout.
- Frontend `signOut()` (Clerk SDK) invalidates the Clerk session but the JWT itself is stateless and verifiable until `exp`.

**Impact:** State transparency violation; user perceives signout but token is still technically valid.
**File/Lines:** `backend/api/deps.py:159-184`, `backend/services/auth_provider.py`.
