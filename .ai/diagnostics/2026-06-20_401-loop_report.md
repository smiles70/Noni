# Diagnostic Report — Deep Diagnosis: 401 → TRANSIENT_ERROR Infinite Loop

**Session:** 2026-06-20 (v4 Agent Harness)
**Agent:** Diagnostic Agent v4
**Status:** COMPLETE — root cause identified, fix applied, verified

---

## 1. Executive Summary

The persistent "looping reloading" issue reported by the user was caused by **all HTTP 401 Unauthorized responses being misclassified as `TRANSIENT_ERROR`**. When `AuthPendingBanner` detects `TRANSIENT_ERROR`, it auto-retries session resolution every 15 seconds. Because a 401 means the token is definitively rejected, each retry fails with another 401, creating an infinite loop that never resolves.

**Fix:** Added an explicit `status === 401` guard in `useAuthSession.ts` `handleError` that transitions to `SIGNED_OUT` instead of `TRANSIENT_ERROR`.

**Result:** The banner unmounts, the user sees the landing page, and can re-authenticate calmly. No page reload. No alarming error message.

---

## 2. Root Cause Analysis

### 2.1 Backend Contract

`backend/api/routes/auth.py:82-85` raises 401s via FastAPI's `HTTPException`:

```python
raise HTTPException(
    status_code=status.HTTP_401_UNAUTHORIZED,
    detail={"error": {"code": code, "message": message or code}},
)
```

FastAPI serializes the `detail` parameter into the response body, producing:

```json
{"detail": {"error": {"code": "auth.no_credential"}}}
```

### 2.2 Frontend Parsing

`frontend/src/auth/useAuthSession.ts:104` (before fix):

```typescript
const code = apiErr?.response?.data?.error?.code;
```

This expects `data.error.code`, but the actual body path is `data.detail.error.code`. Because `data.error` is `undefined`, `code` is always `undefined` for 401s.

### 2.3 State Machine Misfire

When `code` is undefined, `handleError` falls through to:

```typescript
if (!code) {
  setState({ status: "TRANSIENT_ERROR" });
  return;
}
```

`TRANSIENT_ERROR` → `AuthPendingBanner` mounted → countdown → `onRetry()` → `retryNonce` bumped → `useEffect` re-runs → another 401 → repeat forever.

### 2.4 Why the Previous Fix Was Insufficient

The previous session fixed `AuthPendingBanner.tsx` spatial stability (keeping the "Try now" button visible when exhausted, removing "refresh the page" copy). This only addressed the UX symptom when retries were exhausted. The root cause — 401s being misclassified as transient — remained, so the loop continued indefinitely (the banner never reached the exhausted state because each retry reset the countdown).

## 4. Fix Applied

### 4.1 `useAuthSession.ts` — 401 Guard

Added `status === 401` check before body parsing:

```typescript
function handleError(err: unknown) {
  const apiErr = err as
    | { response?: ApiErrorResponse & { status?: number } }
    | undefined;
  const status = apiErr?.response?.status;
  const code = apiErr?.response?.data?.error?.code;

  // 401 Unauthorized is definitive — the token was rejected by the
  // backend. Retrying with the same token can never succeed, so we
  // transition to SIGNED_OUT rather than TRANSIENT_ERROR to prevent
  // an infinite retry loop (see ADR 0024 §B5).
  if (status === 401) {
    setState({ status: "SIGNED_OUT" });
    return;
  }
  // ... rest of error handling unchanged
}
```

**Rationale:** 401 is a terminal auth state per RFC 7235. The client MUST NOT retry the same credentials. Transitioning to `SIGNED_OUT` is the geragogy-friendly outcome: the user sees the calm landing page, not an alarming error banner.

### 4.2 `AuthProvider.test.tsx` — Test Infrastructure

- Added `API_BASE_URL`, `CLERK_PUBLISHABLE_KEY`, `LOG_LEVEL`, `IS_PROD` to env mock (required by `api/client.ts`)
- Renamed `ref` prop to `captureRef` (React treats `ref` as reserved)
- Added URL-aware `fetch` mock (`/auth/config` returns provider parity probe; others return session)
- Used `vi.waitFor` for all async assertions (React 18 defers effects)
- Added regression test: `transitions to SIGNED_OUT on 401 from /auth/session`

## 8. Follow-up: Clerk Desync Login Loop

### 8.1 New Symptom
After deploying the 401 → `SIGNED_OUT` fix, users reported: *"clicking login, it goes back and reloads the login page endlessly."*

### 8.2 Root Cause of New Loop
The initial fix stopped the `TRANSIENT_ERROR` banner loop but introduced a **credential-source desync**:

1. Backend rejects Clerk token → 401
2. `handleError` sets `SIGNED_OUT` (our fix)
3. **But Clerk's SDK still believes `isSignedIn: true`** — we never cleared the Clerk session
4. User clicks "Log in" → `navigate("/signin")`
5. Clerk's `<SignIn routing="virtual" fallbackRedirectUrl="/" />` sees an active Clerk session
6. Clerk auto-redirects the already-signed-in user to `fallbackRedirectUrl="/"`
7. Landing page shows "Log in" because our state is `SIGNED_OUT`
8. User clicks "Log in" again → repeat forever

This is a **redirect loop between `/` and `/signin` driven by contradictory auth state** (Clerk says signed-in, AuthProvider says signed-out).

### 8.3 Additional Finding: Unreadable 401 Bodies
While fixing the desync, we also discovered the frontend was parsing the **wrong JSON path** for all 401 error bodies:

| Source | Actual Body Shape | Frontend Parsed | Result |
|---|---|---|---|
| `backend/api/routes/auth.py` | `{"detail": {"error": {"code": "..."}}}` | `data.error.code` | `undefined` — code unreadable |
| `backend/api/deps.py` | `{"detail": {"envelope_id": "auth.signed_out"}}` | `data.error.code` | `undefined` — code unreadable |

This means **every 401 code was invisible to the frontend**, preventing fine-grained handling (e.g., `auth.account_deleted` should show `REJECTED`, not `SIGNED_OUT`).

### 8.4 Updated Fix

`useAuthSession.ts` changes:

1. **`handleError` is now `async`** and calls `auth.signOut?.()` before `SIGNED_OUT` on 401:
   ```typescript
   if (status === 401) {
     await auth.signOut?.().catch(() => {});
     setState({ status: "SIGNED_OUT" });
     return;
   }
   ```
   This clears Clerk's session (or mock token) so the credential source and state machine agree.

2. **Body parsing now tries three paths** to read the error code:
   ```typescript
   const code =
     apiErr?.response?.data?.error?.code ||
     apiErr?.response?.data?.detail?.error?.code ||
     apiErr?.response?.data?.detail?.envelope_id;
   ```

3. **`ApiErrorResponse` type** updated to include FastAPI's `detail` wrapper.

4. **`auth` parameter type** now includes optional `signOut?: () => Promise<void>`.
